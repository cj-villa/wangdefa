## Context

`financial` 与 `financial-transaction` 两个 tab 中“更新净值”都是触发长耗时接口的入口。当前交互在请求达到前端超时阈值后会中断 loading 并提示失败，实际后端任务可能仍在执行，导致用户重复点击、重复触发与状态混乱。

本次变更聚焦前端交互一致性：对于该类长耗时更新操作，不设置前端超时中断，持续展示 loading，直到接口返回明确结果。

## Goals / Non-Goals

**Goals:**
- 移除两个入口更新净值请求的前端超时限制。
- 在请求期间保持按钮 loading 与防重复点击状态。
- 成功/失败后恢复按钮状态并给出清晰提示。
- 统一两个页面的交互行为与状态管理方式。

**Non-Goals:**
- 不修改后端接口实现与执行时长。
- 不引入轮询、WebSocket、任务队列等新机制。
- 不重构页面整体布局，仅调整更新净值相关交互。

## Decisions

### 1) 更新净值请求采用“无超时”策略
- 决策：更新净值请求使用不设置超时（或超时设为无限）的调用方式。
- 原因：该操作本质为长任务，前端超时会造成误判失败。
- 备选：继续短超时并提示重试。未选原因：会放大重复触发和状态不一致。

### 2) 将 loading 状态绑定到单一请求生命周期
- 决策：点击后进入 loading 并禁用重复触发，只有请求 `finally` 才恢复。
- 原因：确保交互状态与真实请求生命周期一致。
- 备选：定时清除 loading。未选原因：仍可能早于真实完成时间导致误操作。

### 3) 统一两个页面的反馈文案与错误处理
- 决策：成功提示“更新完成”，失败提示明确错误信息；两处实现保持一致。
- 原因：降低学习成本，减少跨页面体验差异。
- 备选：各页面自定义处理。未选原因：长期维护易分叉。

## Risks / Trade-offs

- [风险] 极端情况下请求长期不返回，用户会一直处于 loading  
  → Mitigation: 保持页面可见状态提示，并在后续版本评估“后台执行+轮询”方案。

- [风险] 无超时可能掩盖网络层问题  
  → Mitigation: 失败分支保留错误上报与提示，便于排查网络异常。

- [权衡] 立即性反馈减少，改为等待真实完成  
  → Mitigation: 通过明确的 loading 与防重入降低用户焦虑和误操作。

## Migration Plan

1. 在 `net-value-table.tsx` 调整更新净值请求配置与 loading 生命周期。
2. 在 `financial-transaction.tsx` 做同样调整并统一提示文案。
3. 本地验证两处点击行为：长耗时期间持续 loading、不可重复点击、返回后恢复。
4. 运行前端 `eslint`/`tsc` 及相关页面回归检查。
5. 回滚策略：该变更仅前端交互逻辑，异常时可直接回退对应前端提交。
