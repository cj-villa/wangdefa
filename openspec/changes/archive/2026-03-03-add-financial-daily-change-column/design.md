## Context

当前理财净值列表已展示日期、净值等基础信息，但缺少日涨幅字段与直观涨跌视觉提示。用户需要在列表场景快速判断日表现，因此需要在现有表格结构中增加一列并统一颜色语义。约束是不能影响现有筛选、分页、排序和接口调用流程。

## Goals / Non-Goals

**Goals:**
- 在净值列表新增“日涨幅”列，按百分比显示。
- 按涨跌语义着色：上涨红色、下跌绿色、持平中性。
- 兼容后端可能存在的字段差异（如 `dailyIncreaseRate` / `dailyChangeRate` / `changePercent`）。
- 保持现有交互链路与性能行为不变。

**Non-Goals:**
- 不改动后端接口协议。
- 不重构整个理财列表页面或表格组件。
- 不新增第三方图表或样式依赖。

## Decisions

- 决策 1：在 `net-value-table.tsx` 的列定义中直接新增 `dailyChange` 展示列。
  - 原因：改动范围最小，复用现有 ProTable 列渲染能力。
  - 备选方案：在接口层统一映射再透传。该方案侵入更大、收益有限，本次不采用。

- 决策 2：通过本地格式化函数统一处理日涨幅数值与颜色。
  - 原因：可集中处理空值、字符串数值、百分比格式和颜色判断，降低列渲染复杂度。
  - 备选方案：在 JSX 内联判断。可读性较差且难复用，不采用。

- 决策 3：空值显示 `--`，异常值回退中性样式。
  - 原因：避免异常数据导致页面报错或颜色误导。

## Risks / Trade-offs

- [字段不一致风险] 后端返回字段名可能与预期不同 → 在前端增加多字段兜底映射并保留中性回退。
- [颜色语义差异] 不同业务对红涨绿跌认知可能不同 → 按当前业务明确要求实现，并将颜色常量集中定义便于后续切换。
- [格式风险] 百分比精度不统一 → 固定小数位策略（如 2 位）并在工具函数中统一处理。

## Migration Plan

- 前端灰度发布即可，无需数据迁移。
- 回滚策略：仅回退 `net-value-table.tsx` 与新增格式化代码变更。

## Open Questions

- 日涨幅最终字段优先级是否固定（若后端后续标准化可简化映射逻辑）。
