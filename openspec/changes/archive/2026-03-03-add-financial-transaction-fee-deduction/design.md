## Context

理财模块当前交易成本仅按交易金额处理，未纳入手续费，导致实际净值与收益偏差；同时理财编辑表单缺少手续费配置入口。该变更涉及前后端字段扩展与交易计算逻辑调整，属于跨模块改动（表单、请求类型、后端服务/DTO/实体）。

## Goals / Non-Goals

**Goals:**
- 在理财新增/编辑表单中新增手续费输入项，默认 `0`。
- 前后端数据结构支持手续费字段并保持向后兼容。
- 交易净值计算引入手续费扣减：有效金额 = 交易金额 - 手续费。
- 历史无手续费数据自动按 0 处理。

**Non-Goals:**
- 不调整交易类型（买入/卖出）现有业务语义。
- 不改动非理财模块接口。
- 不引入新的第三方依赖或复杂迁移脚本。

## Decisions

- 决策 1：手续费作为理财字段持久化，默认值为 0。
  - 原因：手续费通常与产品/渠道配置绑定，编辑理财时配置一次更符合当前交互模型。
  - 备选：在每笔交易上单独录入手续费。会显著增加录入复杂度，本次不采用。

- 决策 2：计算层统一使用 `netAmount = amount - fee` 参与份额/净值计算。
  - 原因：将扣费逻辑集中在领域服务，避免前后端重复扣减。
  - 备选：前端预扣后传后端。存在重复扣减和一致性风险，不采用。

- 决策 3：对历史数据使用默认 0（数据库默认值 + 读取兜底）。
  - 原因：避免一次性数据迁移影响上线节奏。

## Risks / Trade-offs

- [字段定义不一致] 前后端字段名或类型不一致导致校验失败 → 统一在 DTO/TS 类型中定义 `fee` 为 number 并默认 0。
- [负值风险] 当金额小于手续费时可能出现负净额 → 增加校验规则，限制手续费不得大于金额或在计算时最小值钳制为 0（按现有业务规则选一种）。
- [历史兼容风险] 历史数据为空导致计算异常 → 在实体默认值和服务读取逻辑双重兜底。

## Migration Plan

- 数据库层为手续费字段设置默认值 0（若字段已存在则补默认策略）。
- 前端发布后新增/编辑理财自动带上手续费字段。
- 回滚时可忽略手续费字段（默认 0 不影响历史流程）。

## Open Questions

- 手续费是否允许小数精度限制（例如最多 2 位）以及是否允许大于交易金额。
