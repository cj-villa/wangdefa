## Context

当前仓库包含前后端多个工作区，ESLint 报错来源分散，既有格式类问题，也有规则约束类问题（如 hooks、import 顺序、未使用变量、类型不安全写法）。现状导致 CI 与本地提交稳定性不足，并且不同开发者对同类问题修复方式不一致。

## Goals / Non-Goals

**Goals:**
- 在不改变业务行为的前提下，修复当前代码库 ESLint 报错到零错误。
- 将修复流程标准化为“自动修复优先、手动修复补齐、全量复检收口”。
- 对典型高频规则形成统一改造策略，降低回归概率。

**Non-Goals:**
- 不引入新业务需求或重构无关模块。
- 不大规模调整 ESLint 规则本身以规避报错。
- 不以牺牲可读性为代价进行机械化修复。

## Decisions

1. 采用分层修复策略：
- 第一层：执行 ESLint 自动修复（`--fix`）处理格式与可自动转换问题。
- 第二层：针对无法自动修复的规则做最小代码改动，保持原有行为。
- 第三层：按工作区执行 lint 全量复检，直到零错误。

备选方案：一次性手改全部问题。未采用原因：效率低、漏改风险高、可重复性差。

2. 统一“行为不变”原则：
- 所有手动修复必须以不改变接口契约和运行时行为为前提。
- 对可能触发行为变化的修复（如 hooks 依赖调整）采用更保守的结构化拆分。

备选方案：允许伴随小范围行为优化。未采用原因：超出本次质量治理范围，验收边界不清晰。

3. 将修复结果纳入交付门禁：
- 变更完成后必须通过 ESLint 全量校验。
- 若项目流程要求，同时补跑 tsc/build 确认无连带回归。

备选方案：只对本次改动文件做 lint。未采用原因：无法实现“修复所有报错”的目标。

## Risks / Trade-offs

- [修复范围过大导致周期增长] → 先按包/模块分批清理并持续复检，逐步收口。
- [规则修复引入潜在回归] → 保持最小改动，并在关键页面/接口补做快速回归验证。
- [历史代码风格不一致] → 在本次中优先遵循现有 lint 规则，后续再考虑风格统一专项。
- [部分报错来自基础配置或依赖版本] → 记录并单独处理配置层问题，避免与业务代码修复耦合。

## Migration Plan

1. 识别各工作区 lint 命令与报错分布，建立修复清单。
2. 执行自动修复并提交第一轮结果。
3. 对剩余错误按规则类型逐项手动修复。
4. 执行全量 lint（必要时包含 tsc/build）并完成收口。
5. 记录仍需后续治理的非阻塞问题（如警告或性能建议）。

回滚策略：若某类修复引发回归，按提交粒度回退对应改动并采用替代写法重新修复。
