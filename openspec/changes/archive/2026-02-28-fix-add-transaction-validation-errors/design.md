## Context

当前“新增交易”表单在前端提交的数据类型与后端 DTO 校验规则不一致：`amount` 被后端按 `number string` 校验，`transactionDate`/`ensureDate` 被后端按 `Date` 实例校验，导致创建请求失败。问题跨越前端表单、请求序列化和后端 DTO 转换链路。

## Goals / Non-Goals

**Goals:**
- 修复新增交易接口的金额与日期字段类型不匹配问题。
- 保证前端提交格式与后端 DTO 校验/转换规则一致。
- 为交易创建链路增加最小回归验证，避免相同问题再次出现。

**Non-Goals:**
- 不改动交易业务语义（买入卖出计算、净值更新逻辑）。
- 不重构整套金融模块 API 设计。
- 不引入新的创建字段或接口。

## Decisions

1. 以后端 DTO 契约为准，统一前端提交格式。
- `amount` 在提交前显式序列化为字符串数字（例如 `"123.45"`）。
- `transactionDate`/`ensureDate` 在提交前转换为后端可转换的标准日期值（ISO 字符串或时间戳，按 DTO 约束选定一种）。

备选方案：放宽后端校验直接接收任意类型。未采用原因：会降低接口契约清晰度并扩大隐性输入风险。

2. 后端对日期字段采用显式转换与校验组合。
- 在 DTO 中使用 class-transformer 将请求值转换为 `Date`。
- 保留 `IsDate`/相关校验，确保非法日期仍被拒绝。

备选方案：仅在 service 层手动 `new Date()`。未采用原因：校验入口分散，不利于统一错误处理。

3. 端到端补充“有效输入可创建、非法输入报错”的最小验证。
- 覆盖金额为字符串数字、日期为可解析值的成功路径。
- 覆盖金额/日期无效值的失败路径。

## Risks / Trade-offs

- [前端日期格式与后端解析时区差异] → 统一使用明确格式并在 DTO 层做一致转换。
- [历史调用方仍传旧格式] → 保持 DTO 兼容常见可解析日期值，避免破坏已有调用。
- [只修创建链路导致更新链路潜在同类问题] → 在任务中增加关联接口快速巡检。

## Migration Plan

1. 盘点当前创建交易请求 payload 与 DTO 校验规则。
2. 调整前端提交序列化逻辑并联调。
3. 调整后端 DTO 转换/校验配置并验证错误信息。
4. 运行 lint/tsc/build 与最小接口回归。

## Open Questions
