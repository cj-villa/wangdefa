## Context

financial 页面目前同时承载“理财列表”和“理财交易”两个核心场景，但筛选维度和交互反馈并未统一：理财列表仍暴露低价值筛选项（昨日收益、余额），交易列表后端筛选能力与前端现有筛选入口不完全对齐。结果是用户在同一业务流程（选产品 -> 看交易）中需要重复操作，且筛选结果可预测性不足。

## Goals / Non-Goals

**Goals:**
- 收敛理财列表筛选项，移除“昨日收益/余额”筛选入口，降低筛选噪音。
- 为理财交易列表接口补齐并对齐前端现有筛选参数，确保“筛选可用、结果一致”。
- 优化 financial 页面交互：筛选状态可见、清空路径清晰、切页/切 tab 时行为可预期。
- 统一加载/空态/错误态的操作反馈，减少用户中断。

**Non-Goals:**
- 不调整理财收益计算逻辑与数据模型。
- 不新增复杂查询语法（如高级组合条件、保存筛选模板）。
- 不改动 financial-detail 的图表计算规则。

## Decisions

1. 筛选项“减法优先”
- 决策：理财列表仅保留名称、编码、渠道等核心筛选项，移除昨日收益、余额筛选。
- 原因：收益/余额本身实时波动且更适合作为结果展示，不适合常用筛选入口。
- 备选：保留并折叠到高级筛选。未选原因：交互复杂度仍偏高，收益有限。

2. 交易列表接口与前端筛选一一对齐
- 决策：后端 `financial/transaction/list` 查询对象扩展支持现有页面筛选字段（如 code/name/transactionType/financialId 等），控制器与 service 层同步透传。
- 原因：避免“UI 有筛选控件但接口不生效”的体验断层。
- 备选：仅前端本地过滤。未选原因：分页数据不完整，结果不可靠。

3. 统一筛选交互约定
- 决策：采用“显式应用 + 一键清空 + 可视化筛选状态（tag/文案）”模式。
- 原因：便于用户理解当前列表状态，减少误判。
- 备选：输入即查询。未选原因：高频请求与状态抖动更明显。

4. 状态反馈标准化
- 决策：列表与交易页统一使用加载骨架、空状态提示、错误重试入口，且保留当前筛选上下文。
- 原因：保证异常时可恢复，避免用户重复输入筛选条件。

## Risks / Trade-offs

- [Risk] 交易列表查询条件扩展后可能影响旧查询性能 -> [Mitigation] 仅增加必要索引字段过滤并保持分页默认限制。
- [Risk] 前后端筛选字段命名不一致导致回归 -> [Mitigation] 在 request type 与 controller query 中统一命名并补充验证。
- [Trade-off] 列表移除收益/余额筛选降低了“极少数高级筛选”能力 -> [Mitigation] 保留列排序与结果高亮作为替代路径。

## Migration Plan

1. 前端先完成筛选 UI 调整与参数映射。
2. 后端扩展交易列表 query 与接口文档，保持兼容旧参数。
3. 联调验证“筛选-分页-重置-切 tab”全链路。
4. 回归检查后上线；如出现筛选回归，回滚到旧 query 处理逻辑。

## Open Questions

- 交易列表是否需要新增“渠道”筛选（取决于业务是否高频使用）。
- 是否在后续迭代增加“保存最近筛选条件”能力。
