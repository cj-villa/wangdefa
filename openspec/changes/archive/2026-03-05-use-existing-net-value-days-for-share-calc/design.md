## Context

当前交易份额计算依赖净值查询逻辑，回溯时按“自然日减 N 天”取净值。该策略在非交易日、节假日或净值补录延迟场景下会命中缺失日期，导致份额计算失败或偏差。改动涉及交易服务与净值服务两个模块，需要先明确统一的“有效净值日”口径，避免各处实现不一致。

## Goals / Non-Goals

**Goals:**
- 将份额计算净值回溯策略统一为“向前取存在净值记录的 N 天”。
- 在净值缺失连续出现时提供可预测的回退行为（继续向前查找，直到命中或无数据）。
- 保持现有接口入参和返回结构兼容。
- 为关键边界场景补充测试，确保回归可控。

**Non-Goals:**
- 不调整交易手续费、金额、份额等业务字段定义。
- 不改造净值抓取流程与数据落库机制。
- 不新增外部依赖与数据库表结构。

## Decisions

1. 在净值查询服务中新增“按有效净值日偏移”的查询方法。
- 方案：以交易生效日期为上界，按日期倒序查询净值记录，跳过缺失日期，取第 N 条记录。
- 备选：在交易服务中循环按自然日回退并逐日查询。未采用原因：查询次数不可控且逻辑分散。

2. 交易份额计算统一依赖新方法，不再直接按自然日偏移计算目标日期。
- 方案：在创建/更新交易份额时调用净值服务新方法获取目标净值。
- 备选：保留旧逻辑并在失败时兜底新逻辑。未采用原因：双路径会导致行为不一致。

3. 缺失数据处理采用“显式失败”策略。
- 方案：若在可查询范围内仍找不到满足偏移条件的净值，返回明确业务异常，阻止错误份额入库。
- 备选：默认使用最近一条净值。未采用原因：会掩盖数据缺口并引入静默误差。

## Risks / Trade-offs

- [历史数据缺口较大导致交易保存失败率上升] → 在错误信息中明确提示“有效净值不足”，便于补录后重试。
- [查询策略变化导致历史行为差异] → 增加单测覆盖旧场景与缺失场景，确保预期变更可验证。
- [多模块改动引入回归] → 仅在净值服务集中实现核心逻辑，交易服务只做调用替换，降低扩散面。

## Migration Plan

1. 在净值服务实现“按有效净值日偏移”查询能力。
2. 替换交易创建/更新时的净值获取调用。
3. 增加并运行相关单测、`eslint`、`tsc`。
4. 上线后观察交易保存异常日志，确认缺失净值提示符合预期。

## Open Questions

- “N 天”是否始终等于当前 financial.delay，还是后续允许按交易类型配置不同偏移？
- 当净值数据仅缺失少量天数时，是否需要在响应中返回实际命中的净值日期供前端提示？
