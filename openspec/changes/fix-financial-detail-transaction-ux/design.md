## Context

理财模块在列表、详情、交易三个页面之间存在数据边界与交互边界不一致问题：详情页关键指标混入全局汇总值；列表页“查看交易”未正确切换到交易视图；交易筛选同时暴露文本字段（基金名称/编码）与实际业务语义不一致，导致筛选复杂且易错。当前系统为 TypeScript + React 前端配合 Node.js 接口，需要在不破坏现有路由结构的前提下修复数据来源和页面联动。

## Goals / Non-Goals

**Goals:**
- 详情页关键指标严格按当前基金维度展示。
- 从理财列表进入交易视图时，导航到正确 Tab，并保留目标基金上下文。
- 交易筛选项精简为基金选择器，移除冗余字段，并保证筛选参数与接口一致。
- 保持现有业务接口形态兼容，确保改动后 eslint/tsc 可通过。

**Non-Goals:**
- 不重构整套理财路由架构。
- 不引入新的后端存储模型。
- 不改变与本次 bug 无关的收益计算口径。

## Decisions

- 基金详情数据来源按“当前基金 ID”做约束读取，禁止复用页面级聚合数据对象。
  - 备选：在前端按总数据二次过滤。
  - 取舍：优先在数据源层面保证正确性，避免前端补丁式计算引入偏差。

- “查看交易”采用显式路由参数或 query 标记目标 Tab 与基金标识，进入页面后统一由交易页初始化逻辑消费。
  - 备选：通过全局状态临时传值。
  - 取舍：路由可回放、可分享、可刷新恢复，行为更稳定。

- 交易筛选改为单一基金选择器（value 为基金主键/代码），移除基金名称与编码输入框。
  - 备选：保留输入框并做联动自动补全。
  - 取舍：单一入口减少歧义，降低筛选参数错配概率。

- 交易列表请求参数映射统一在 service 层做适配，页面层只维护语义化筛选状态。
  - 备选：页面直接拼装接口参数。
  - 取舍：集中适配便于后续接口字段调整和单测覆盖。

## Risks / Trade-offs

- [Risk] 路由参数变更可能影响已有深链访问 → Mitigation：兼容旧参数解析，新增参数走兜底。
- [Risk] 基金选择器数据源延迟导致初始筛选不可用 → Mitigation：增加加载态与默认空值容错。
- [Risk] 详情数据修复后与历史截图不一致引发疑问 → Mitigation：在发布说明中标注“修复为基金维度数据”。
