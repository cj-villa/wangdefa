## Context

`smart-house-be` 的接口层分布在多个 controller 模块，当前单测覆盖不均衡，部分接口缺少成功路径、参数校验失败路径、依赖服务异常路径的自动化验证。由于接口层是前后端联调与线上排障的第一层边界，测试缺口会在需求迭代时放大回归风险。

本次变更是跨模块测试基线建设，重点是统一接口单测结构与覆盖标准，并将其纳入质量门禁。

## Goals / Non-Goals

**Goals:**
- 为接口控制器建立统一单测模式（controller 单测 + DTO 校验相关测试）。
- 覆盖每个接口最小必要场景：成功、请求参数非法、依赖异常。
- 为后续新增接口提供可复用测试夹具，降低新增测试成本。
- 确保新增测试可稳定通过 `eslint`、`tsc`、`jest`。

**Non-Goals:**
- 不修改现有业务接口协议与返回结构。
- 不引入端到端测试替代方案（本次聚焦单元测试）。
- 不做覆盖率“刷数字”式改造，优先保证关键行为可回归。

## Decisions

### 1) 以接口模块为单位建立测试文件组织
- 决策：按 controller 维度补测，测试文件与现有 `test/core/**` 结构对齐，模块内统一命名与分组。
- 原因：接口职责边界清晰，便于追踪失败来源和长期维护。
- 备选：按 service 维度间接覆盖接口。未选原因：无法直接验证 controller 参数绑定、异常映射和 HTTP 入口行为。

### 2) 统一使用 Nest TestingModule + service mock
- 决策：controller 单测通过 `@nestjs/testing` 创建模块，依赖服务使用 mock provider。
- 原因：保持测试执行快、隔离强，减少外部依赖导致的不稳定。
- 备选：直接起应用做 e2e。未选原因：启动成本高，定位粒度粗，不符合“接口层单测补齐”目标。

### 3) 定义最小覆盖矩阵并固化到任务清单
- 决策：每个接口至少包含成功/参数错误/依赖异常三类场景；涉及 DTO 转换的接口增加转换边界测试。
- 原因：覆盖核心回归风险点，兼顾成本与收益。
- 备选：仅补成功路径。未选原因：无法防止线上常见异常回归。

## Risks / Trade-offs

- [风险] 一次性补测范围大，短期工作量较高  
  → Mitigation: 分模块分批提交，优先高频与高风险接口。

- [风险] mock 行为与真实运行环境存在偏差  
  → Mitigation: 对关键模块补少量集成校验并保持 mock 与接口契约一致。

- [权衡] 统一模板会牺牲少量测试写法自由度  
  → Mitigation: 保留可扩展夹具入口，允许复杂场景按需增强。

## Migration Plan

1. 盘点 `interface/modules` 中所有 controller 与 endpoint，生成测试补齐清单。
2. 建立统一测试夹具（mock provider、请求样例、断言 helper）。
3. 按模块补齐测试并逐步执行本地校验。
4. 在 CI 中确保新增测试纳入常规校验流程。
5. 回滚策略：该变更仅新增测试与少量测试辅助代码，不影响生产运行；若出现问题可回退对应测试提交。

## Open Questions

- 是否为“全量接口”定义阶段性范围（例如先 financial/firefly，再扩展全项目）以避免单次变更过大？
- 是否需要在仓库层引入覆盖率阈值（按模块/全局）作为后续强制门禁？
