## Context

当前项目存在多个分页入口（表格、列表页、查询结果页），每个页面对 `pageSize` 可选项配置不一致，部分仅支持单一值或与其他页面不一致。该问题跨越多个前端页面与通用列表配置，属于横向一致性改造。

## Goals / Non-Goals

**Goals:**
- 将分页大小可选项统一为 `10`、`20`、`50`。
- 对齐默认分页大小与分页切换行为，避免不同页面表现不一致。
- 保持现有分页参数协议不变（`current`/`pageSize`），确保后端兼容。
- 增加关键页面回归验证，确保切换分页大小后请求参数和展示正确。

**Non-Goals:**
- 不改动分页接口字段命名和响应结构。
- 不引入新的分页交互模式（如无限滚动）。
- 不重构与分页无关的页面筛选或排序逻辑。

## Decisions

1. 统一分页大小常量在前端集中维护。
- 方案：抽取 `PAGE_SIZE_OPTIONS = [10, 20, 50]` 并在分页组件/配置复用。
- 备选：逐页面硬编码。未采用原因：后续维护成本高且易再次分叉。

2. 逐页面替换为统一配置，不调整 API 协议。
- 方案：保持请求参数 `pageSize` 透传，页面只改可选项和默认值。
- 备选：后端改默认值后前端不改。未采用原因：无法保证用户可选项一致。

3. 为关键列表补充行为验证。
- 方案：覆盖分页大小切换后请求参数、页码重置/保持策略与数据刷新。
- 备选：仅人工验证。未采用原因：回归风险较高。

## Risks / Trade-offs

- [部分页面未接入统一分页组件] → 通过全局检索 `pageSize` 与分页配置点逐一清单化改造。
- [默认页大小变化影响用户习惯] → 默认保持现状，仅统一可选项为 `10/20/50`。
- [分页状态管理差异导致行为不一致] → 明确约束切换 `pageSize` 后触发一次标准列表请求并更新分页状态。

## Migration Plan

1. 盘点前端所有分页入口并统一配置来源。
2. 批量替换分页可选项为 `10/20/50`。
3. 验证关键页面分页请求参数与展示行为。
4. 执行 `eslint`、`tsc` 与相关测试。

## Open Questions

- 是否需要将默认值也统一为 `10`，还是维持各页面当前默认值仅统一可选项？
- 历史页面中存在自定义分页组件时，是否本次一并替换为通用组件？
